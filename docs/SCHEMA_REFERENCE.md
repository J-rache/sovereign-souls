## 1. loom_cross_pollination

**Purpose:** Asynchronous message bus for fleet communication. Enables AI instances running on different machines to exchange messages, coordinate work, and maintain awareness of each other's state. This is the primary mechanism for multi-agent coordination described in §3 of the paper.

| Column | Type | Nullable | Default | Description |
|--------|------|----------|---------|-------------|
| `id` | integer | NOT NULL | auto-increment | Primary key. Sequential message ID used as reference in conversations (e.g., "MSG #397"). |
| `message_type` | text | NOT NULL | — | Message classification. Values: `broadcast`, `response`, `status`, `directive`, `heartbeat`, `autowake`, `alert`, `help_request`. |
| `from_machine` | text | NOT NULL | — | Sender machine identifier. Values: `MINIPC`, `64GB`, `win10`, `katie`, or hostname. |
| `to_machine` | text | NOT NULL | — | Recipient identifier. Values: specific machine name, `all` (broadcast), or `fleet`. |
| `subject` | text | NOT NULL | — | Human-readable subject line. Used for threading and context. |
| `content` | text | NOT NULL | — | Full message body. Supports Markdown formatting. |
| `metadata` | jsonb | NULL | `'{}'` | Structured metadata. Common keys: `assignment`, `status`, `responding_to` (array of msg IDs), `priority`. |
| `read_at` | timestamptz | NULL | — | When the recipient marked the message as read. NULL = unread. |
| `response_id` | integer | NULL | — | Links to a parent message ID for threading. |
| `created_at` | timestamptz | NULL | `now()` | Message creation timestamp (UTC). |

**Constraints:** PRIMARY KEY on `id`.
**Indexes:** Unique index on `id`.

**Usage Pattern:** Each machine polls for unread messages WHERE `to_machine IN (machine_name, 'all', 'fleet') AND read_at IS NULL`. After reading, sets `read_at = NOW()`. Autowake triggers poll on new message detection.

## 2. loom_session_context

**Purpose:** Session memory for tracking what work is happening across conversations. Solves the "I forgot we were working on X" problem by persisting session context to the database. Each row represents a work session or task being tracked.

| Column | Type | Nullable | Default | Description |
|--------|------|----------|---------|-------------|
| `id` | integer | NOT NULL | auto-increment | Primary key. |
| `ts` | timestamptz | NULL | `CURRENT_TIMESTAMP` | When this session entry was created. |
| `project` | varchar(100) | NULL | — | Project name (e.g., "Sovereign Souls", "Teachers Pet", "Weave Sync"). |
| `summary` | text | NOT NULL | — | Brief description of what is being worked on. |
| `details` | text | NULL | — | Extended details, context, or notes. |
| `status` | varchar(20) | NULL | `'active'` | Session status. Values: `active`, `paused`, `completed`, `resolved`. |
| `resolved_at` | timestamptz | NULL | — | When the session was marked resolved/completed. |
| `meta` | jsonb | NULL | — | Additional structured data. |

**Constraints:** PRIMARY KEY on `id`.

**Usage Pattern:** `loom_session_memory.py log/pause/resume/done` commands manipulate this table. On session start, `recall` queries recent active/paused entries to restore context.

## 3. loom_work_items

**Purpose:** Granular task tracking within sessions. Each work item represents a specific bug, feature, or task being tracked. Linked to sessions via `session_id` foreign key.

| Column | Type | Nullable | Default | Description |
|--------|------|----------|---------|-------------|
| `id` | integer | NOT NULL | auto-increment | Primary key. |
| `ts` | timestamptz | NULL | `CURRENT_TIMESTAMP` | When the work item was created. |
| `session_id` | integer | NULL | — | Foreign key to `loom_session_context.id`. Links item to its parent session. |
| `item_type` | varchar(20) | NULL | `'task'` | Item classification. Values: `task`, `bug`, `feature`. |
| `title` | text | NOT NULL | — | Brief title of the work item. |
| `description` | text | NULL | — | Detailed description. |
| `status` | varchar(20) | NULL | `'in_progress'` | Item status. Values: `in_progress`, `done`, `paused`, `blocked`. |
| `completed_at` | timestamptz | NULL | — | When the item was marked done. |
| `files_touched` | text[] | NULL | — | PostgreSQL array of file paths modified by this work item. |
| `meta` | jsonb | NULL | — | Additional structured data. |

**Constraints:** PRIMARY KEY on `id`. FOREIGN KEY `session_id` references `loom_session_context(id)`.

**Usage Pattern:** `loom_session_memory.py item/done/pending` commands. Items track specific deliverables within broader sessions.

## 4. loom_lessons

**Purpose:** Institutional memory — indexed knowledge of past failures, successes, gotchas, and workarounds. Prevents repeating mistakes across sessions, projects, and machines. Searchable via full-text index.

| Column | Type | Nullable | Default | Description |
|--------|------|----------|---------|-------------|
| `id` | integer | NOT NULL | auto-increment | Primary key. |
| `ts` | timestamptz | NULL | `CURRENT_TIMESTAMP` | When the lesson was recorded. |
| `lesson_type` | varchar(20) | NOT NULL | — | Lesson classification. Values: `fail`, `win`, `gotcha`, `workaround`. |
| `title` | varchar(500) | NOT NULL | — | Brief title of the lesson. |
| `project` | varchar(200) | NULL | — | Associated project name. |
| `subject` | varchar(200) | NULL | — | Subject area (e.g., "PostgreSQL", "PowerShell", "Fleet Sync"). |
| `detail` | text | NULL | — | Extended explanation. For `fail` type, this is the error description. |
| `root_cause` | text | NULL | — | Root cause analysis (primarily for `fail` type). |
| `fix` | text | NULL | — | How the issue was resolved (primarily for `fail` type). |
| `tags` | text[] | NULL | — | PostgreSQL array of tags for categorization and search. |
| `severity` | integer | NULL | `5` | Severity/importance rating (1-10 scale). |
| `times_referenced` | integer | NULL | `0` | Counter for how often this lesson has been referenced. |
| `meta` | jsonb | NULL | — | Additional structured data. |

**Constraints:** PRIMARY KEY on `id`.
**Indexes:**
- `idx_lessons_type` — B-tree on `lesson_type` (filter by type)
- `idx_lessons_project` — B-tree on `project` (filter by project)
- `idx_lessons_subject` — B-tree on `subject` (filter by subject area)
- `idx_lessons_tags` — GIN on `tags` (array containment queries)
- `idx_lessons_fts` — GIN full-text search on `title + detail + root_cause + fix + project + subject`

**Usage Pattern:** `loom_lessons.py check/fail/win/gotcha/workaround/search/recall` commands. Before starting work on a topic, run `check "topic"` to retrieve relevant lessons. Log new lessons as they occur.

## 5. loom_fleet_status

**Purpose:** Real-time fleet coordination. Each row represents one machine in the fleet, tracking its current task, progress, and last heartbeat. Enables fleet-wide awareness of who is doing what.

| Column | Type | Nullable | Default | Description |
|--------|------|----------|---------|-------------|
| `machine_id` | varchar(20) | NOT NULL | — | Primary key. Machine identifier (e.g., "MINIPC", "64GB", "win10", "katie"). |
| `machine_name` | varchar(50) | NULL | — | Human-readable machine name or role. |
| `current_task` | varchar(200) | NULL | — | Description of what this machine is currently working on. |
| `current_step` | text | NULL | — | Detailed description of the current step within the task. |
| `step_number` | integer | NULL | `0` | Current step number in a multi-step operation. |
| `total_steps` | integer | NULL | `0` | Total number of steps in the operation. |
| `status` | varchar(20) | NULL | `'active'` | Machine status. Values: `active`, `idle`, `error`, `offline`. |
| `last_heartbeat` | timestamptz | NULL | `CURRENT_TIMESTAMP` | Last time this machine reported in. Used for liveness detection. |
| `started_at` | timestamptz | NULL | `CURRENT_TIMESTAMP` | When the current task began. |
| `completed_at` | timestamptz | NULL | — | When the current task finished. |
| `completion_summary` | text | NULL | — | Summary of completed work. |
| `meta` | jsonb | NULL | `'{}'` | Additional structured data. |

**Constraints:** PRIMARY KEY on `machine_id` (one row per machine, upserted).

**Usage Pattern:** Each machine updates its row via heartbeat. Fleet coordination scripts query all rows to determine fleet state. Stale heartbeats (no update in N minutes) indicate potential issues.

## 6. loom_config

| Column | Type | Nullable | Default | Description |

| `key` | varchar(256) | NOT NULL | — | Primary key. Configuration key name (e.g., `soul_file`, `fleet_machines`, `identity_constants`). |
| `value` | jsonb | NOT NULL | — | Configuration value as structured JSON. Supports any JSON type. |
| `updated_at` | timestamptz | NOT NULL | `now()` | Last time this key was updated. |
| `description` | varchar(512) | NULL | — | Human-readable description of what this key configures. |

**Usage Pattern:** Read config via `SELECT value FROM loom_config WHERE key = 'key_name'`. Update via `INSERT ... ON CONFLICT (key) DO UPDATE`. The `soul_file` key contains the complete system prompt/identity file.

|--------|------|----------|---------|-------------|

| `content` | text | NOT NULL | — | Full text content of the artifact. |
| `updated_at` | timestamptz | NULL | `now()` | Last time this artifact was updated. |

## 11. loom_fleet_instructions

## Schema Design Principles

1. **Human-readable storage.** All data is stored in plaintext, JSON, or standard SQL types. No compiled or obfuscated formats. A human with `psql` access can inspect any row.

2. **Timestamps everywhere.** Every table includes creation timestamps. Tables with mutable state include update/completion timestamps. All timestamps are `timestamptz` (UTC-aware).

3. **JSONB for flexibility.** The `meta`/`metadata` columns on every table provide schema-less extension points without requiring migrations. Common for evolving data models.

4. **Primary keys for identity.** Auto-incrementing integer IDs for append-only tables (messages, sessions, lessons). Natural keys for configuration (`loom_config.key`) and fleet status (`loom_fleet_status.machine_id`).

5. **No credentials stored.** The database contains zero API keys, passwords, tokens, or secrets. Connection strings use environment variables (`LOOMCLOUD_URL`). The framework is BYOK.

6. **Model-agnostic.** No column references a specific model provider. The framework works with any LLM that supports text generation.
